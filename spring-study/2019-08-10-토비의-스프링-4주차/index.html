스프링 스터디 4주차 내용정리 - 예외

Sat, Aug 10, 2019

1. 초난감 예외처리

부적절한 예외처리

  • 예외 블랙홀
    • 예외를 잡고 아무런 처리를 하지 않는 경우.
    • 예외를 무시하고 계속 진행하여, 후에 다른 오류가 발생했을때, 어떤 원인으로 발생하였는지 알기 어려움.

  • 습관적인 예외 처리 회피문제

    • method의 선언부에 발생하는 예외를 습관적으로 회피하는 코드를 작성(throws Exception)

    • 실제로 적절한 처리를 통해 복구 될 수 있는 경우도, 습관적인 예외회피와 구분이 힘들어서 기회를 박탈당하는 문제 발생.

      예외 처리의 원칙

  • 예외는 적절하게 복구 되어야 한다.

  • 복구 될 수 없는 경우라면, 작업을 중단시키고 개발자에게 알려야한다.

2. 예외 처리의 개선

예외의 분류 의도 이해

  • Error 시스템에 비정상적인 상황이 발생했을 때 사용.
  • Exception 개발자의 코드를 실행하는 중에 발생하는 오류에 대해서 발생하도록 의도.
    • 체크 예외 예상하지 못한 예외 상황에서 발생하도록 의도.
    • 언체크 예외 개발자의 부주의가 원인이 되고, 피할 수 있는 경우에 발생하도록 의도.
  • 분류에 대한 비판
    • 아무런 처리를 할 수 없는 경우에도, 예외처리를 강제하여 무분별한 예외 코드가 작성되도록 한다.

예외 처리 방법 개선

예외 처리 방법의 이해
  • 예외의 처리
  • 예외의 회피

  • 예외의 전환

    • low level 의 예외를 상위 계층에서 의미가 분명하게 전달되도록 전환.

    • 코드를 간결하게 하기 위해 체크 예외를 언체크 예외로 전환.

      예외 처리 전략

  • 지향점

    • 예외를 효과적으로 사용.
    • 예외가 발생하는 경우의 코드를 깔끔하게 정리.

  • 런타임 예외(언체크 예외)의 보편화

    • 자바 어플리케이션의 환경 변화
      • 독립형 어플리케이션 환경에서 서버환경으로 변화.
      • 통제 불가능한 예외에 대해서 대응해주어야할 필요성 감소.
      • 실질적으로 대응가능한 방법 없음.
    • 항상 복구할 수 있는 예외가 아닌 경우 언체크 예외로 만듦.
    • 언체크 예외는 복구 할 수 있는 경우에 대해서도 대응가능.

  • 어플리케이션 예외

    • 어플리케이션 로직 상의 예외.
    • 항상 적절하게 관리 될 필요.
    • 체크 예외로 만들어 처리를 강제.

3. 예외 전환의 활용

데이터 액세스 코드의 추상화와 예외 처리.

데이터 베이스의 종류에 독립적인 dao

  • jdbc 의 한계점과 해결
    • 비표준 SQL
      • 데이터 베이스의 종류에 따라 달라지는 비표준 SQL 문제
      • SQL을 dao에서 독립시키고 데이터베이스 종류에 따라 교체하는 방법으로 해결.
    • 같은 상황에 대한 다른 에러코드
      • 데이터베이스의 종류에 따라 같은 상황에 다른 에러코드 전달.
      • 동일 상황에 대해 표준화 된 상태정보 사용으로 해결.
      • 데이터 베이스의 오류코드를 정리 해놓고 상황에 맞는 예외에 매핑하는 방식으로 해결.

데이터 접근 기술의 종류에 독립적인 dao

  • 전략패턴을 활용하는 경우에 발생하는 문제
    • dao를 구현하는 기술의 종류에 따라 다른 예외를 발생.
    • 체크 예외의 경우 dao 인터페이스의 선언에 구현에서 발생하는 예외를 포함하도록 강제.
    • 사용 기술을 추상화 하는 것이 불가능
  • 문제 해결
    • 데이터 접근 과정에서 발생하는 예외를 언체크 예외로 전환.
    • 복구 가능한 상황에 대한 대응
      • 서로 다른 기술에서 발생하는 예외를 의미를 기준으로 분류하여 추상화.
      • 예외의 의미에 따라 대응가능하도록 분류,
